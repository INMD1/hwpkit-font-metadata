# 자주 하는 수정 요리책

"이런 걸 하고 싶은데 어디를 고쳐야 하지?"에 대한 답을 모았습니다. 각 항목은
"고칠 파일 → 함수 → 빌드/테스트" 순서로 적었습니다.

모든 항목 공통: `.ts` 파일을 고친 뒤에는 항상 `npm run build`로 다시 컴파일하고
`npm test`를 돌려야 합니다 ([00-시작하기.md](./00-시작하기.md) 참고).

---

## 1. 측정할 문자 집합(coverage set)을 추가/변경하고 싶다

예: "확장 한자도 coverage에 넣고 싶다."

1. `src/constants.ts`의 `COVERAGE_SETS` 객체(38번째 줄)에 새 항목을 추가합니다.
   `codePointRange(시작, 끝)`이나 `uniqueCodePoints("문자열")` 헬퍼를 그대로 쓸 수
   있습니다.
2. 폭(advance)도 같이 통계 내고 싶다면 `ADVANCE_GROUPS`(63번째 줄)에도 같은 이름으로
   추가합니다. `COVERAGE_SETS`는 "글자가 있는지"만, `ADVANCE_GROUPS`는 "폭이 얼마인지"
   통계를 냅니다 — 둘은 별개 객체라서 따로 등록해야 합니다.
3. `src/analyze.ts`의 `extractCoverage()`/`extractAdvanceMetrics()`는 이 두 객체를
   `Object.entries()`로 순회하기 때문에 **코드를 더 고칠 필요 없이** 새 집합이 결과
   JSON의 `coverage.sets.<이름>`과 `metrics.advance.groups.<이름>`에 자동으로 나타납니다.
4. `schema/font-catalog.schema.json`에 필드 이름이 하드코딩돼 있다면 같이 갱신하세요
   (스키마가 `additionalProperties`를 막고 있는지 확인).
5. `test/analyze.test.mjs`의 기존 테스트를 참고해서 새 집합이 기대한 대로 잡히는지
   테스트를 하나 추가합니다.

## 2. 원본과 후보를 비교할 때 가중치(중요도)를 바꾸고 싶다

예: "세로(vertical) 차이를 지금보다 더 중요하게 보고 싶다."

1. `src/compare.ts`의 `DEFAULT_COMPONENT_WEIGHTS`(12번째 줄)에서 `width/space/vertical/
   style` 네 값을 조정합니다. 네 값의 합이 1이 되도록 맞추는 게 관례입니다(꼭 강제는
   아니지만 `score` 스케일이 흔들리지 않게 하려면 유지하는 게 좋습니다).
2. 폭 안에서 한글/한자/라틴 중 뭘 더 중요하게 볼지는 `DEFAULT_WIDTH_METRIC_WEIGHTS`
   (19번째 줄), 세로는 `DEFAULT_VERTICAL_METRIC_WEIGHTS`(30번째 줄)에서 조정합니다.
3. 기본값 대신 실행할 때마다 다른 가중치를 쓰게 하려면, `ComparisonOptions`
   (158번째 줄)로 이미 주입 경로가 열려 있습니다. `bin/hwpkit-font-meta.ts`의
   `runCompare()`에서 `rankFontCandidates(sourceProfile, candidates, options)`처럼
   세 번째 인자로 넘기고, CLI 옵션(`--width-weight` 등)을 새로 파싱해서 채우면 됩니다
   — CLI 옵션을 추가하는 방법은 아래 4번 항목을 참고하세요.
4. `test/compare.test.mjs`에 가중치 변경이 점수 순서를 어떻게 바꾸는지 확인하는
   테스트를 추가하는 걸 권장합니다.

## 3. 문서에 자주 나오는 예문(corpus)을 기본값에 추가하고 싶다

예: "표 안에 들어가는 짧은 문장도 기본으로 측정하고 싶다."

1. `src/constants.ts`의 `DEFAULT_LAYOUT_SAMPLES`(140번째 줄)에 `{ id, normalization, text }`
   객체를 추가합니다. `id`는 다른 샘플과 겹치지 않는 이름이면 됩니다.
2. 이 배열이 바뀌면 `corpusFingerprint()`(`src/analyze.ts` 52번째 줄)가 계산하는
   `generator.corpusSha256` 값도 달라집니다. 즉 **기존에 만들어둔 카탈로그 JSON과
   새 카탈로그는 corpus가 달라진 것으로 취급**됩니다. README의 "corpus를 생략하면"
   문단에 이 사실이 이미 안내돼 있으니, 필요하면 `CORPUS_ID`(`src/constants.ts` 7번째
   줄)의 버전 숫자도 함께 올려서 소비자가 구분할 수 있게 하세요(`hwpkit-ko-layout-v1`
   → `v2`).
3. 사용자가 직접 파일로 넣는 corpus(`--corpus`)는 이 상수를 거치지 않고
   `bin/hwpkit-font-meta.ts`의 `loadCorpusSamples()`가 별도로 처리합니다
   ([01-analyze-작동원리.md](./01-analyze-작동원리.md) 참고). 사용자 corpus 동작을
   바꾸고 싶다면 그 함수를 보세요.

## 4. CLI 옵션을 새로 추가하고 싶다

예: "`analyze`에 `--pretty-off` 같은 새 플래그를 추가하고 싶다."

`bin/hwpkit-font-meta.ts`의 옵션 파싱은 세 종류로 나뉩니다.

- `analyze`/`compare` 공통 옵션(`-o`, `--corpus`, `--compact`, `--strict`,
  `--max-files`, `--max-file-size-mib`)은 `parseCommonOption()`(117번째 줄)에서 처리하고
  `BaseOptions` 인터페이스(79번째 줄)에 필드로 있습니다.
- `analyze`만의 옵션(`--face`)은 `parseAnalyze()`(157번째 줄) 안에서 직접 처리합니다.
- `compare`만의 옵션(`--source`, `--top` 등)은 `parseCompare()`(184번째 줄)에서
  처리합니다.
- `merge`만의 옵션은 `parseMerge()`(496번째 줄, [03번 문서](./03-출력관리와-merge.md)
  참고)에서 처리합니다.

새 옵션을 추가하는 순서:

1. 해당하는 `*CliOptions` 인터페이스에 필드를 추가합니다.
2. 해당 `parse*()` 함수 안의 `if (arg === "--내옵션") { ... }` 분기를 추가합니다. 값이
   필요한 옵션이면 `optionValue(args, index, arg)`를 쓰고 `index += 1`을 잊지 마세요.
3. `run*()` 함수(`runAnalyze`/`runCompare`/`runMerge`)에서 그 옵션 값을 실제로
   사용하도록 연결합니다.
4. `HELP` 문자열(29번째 줄 근처)에 새 옵션 설명을 한 줄 추가합니다 — 이걸 빼먹으면
   `--help`로는 아무도 새 옵션을 알 수 없습니다.
5. `test/cli.test.mjs`의 `parseAnalyze`/`parseCompare` 테스트 패턴을 그대로 따라
   새 옵션이 올바르게 파싱되는지 테스트를 추가합니다.

## 5. `-o`를 안 줬을 때 저장 위치/파일명을 바꾸고 싶다

[03-출력관리와-merge.md](./03-출력관리와-merge.md)의 "`result/` 폴더 이름이나 파일명
규칙을 바꾸고 싶다면" 절을 보세요. `RESULT_DIR` 상수와 `defaultOutputPath()` 함수
두 곳만 고치면 됩니다.

## 6. Hwpkit 본체 연결부를 확인/수정하고 싶다

이 저장소는 Hwpkit 본체를 직접 고치지 않지만, 어댑터 예제가 있습니다.

- `examples/hwpkit-consumer.mjs`: 이 저장소가 만든 카탈로그 JSON을 읽어서 `codePointWidthHwp()`,
  `lineStartPositionsHwp()` 같은 함수로 감싸주는 예제입니다. `hwpkit.widthModel`/
  `hwpkit.lineModel`의 필드 이름을 바꿨다면 이 파일도 반드시 함께 고쳐야 합니다.
- Hwpkit 본체 쪽에서 실제로 이 adapter를 연결해야 하는 지점은
  README의 "Hwpkit 연결 예" 절에 세 곳(`HwpEncoder.ts`, `HwpxEncoder.ts` 두 함수)이
  명시돼 있습니다. 이 저장소에서는 수정하지 않는 파일들이니, `hwpkit` 필드의 의미를
  바꿀 때는 그쪽 담당자와 먼저 상의하세요.

## 7. 새 폰트 컨테이너 확장자(예: `.ttf` 외 새 포맷)를 지원하고 싶다

1. `src/constants.ts`의 `FONT_EXTENSIONS`(9번째 줄)에 확장자를 추가합니다.
2. `src/font-source.ts`의 `SUPPORTED_FONT_EXTENSIONS`(8번째 줄)에도 똑같이 추가합니다
   (현재 이 두 목록은 별도로 유지되고 있으니 하나만 고치면 나머지 절반이 그 포맷을
   놓칩니다).
3. `detectFontContainer()`(`src/font-source.ts` 257번째 줄)가 파일의 매직 바이트로
   컨테이너 종류를 알아내는 곳입니다. `fontkit`이 그 포맷을 지원하지 않으면 여기서
   시그니처를 추가해도 실제로 열리지는 않으니, 먼저 `fontkit`이 해당 포맷을 파싱할 수
   있는지 확인하세요.
4. `test/font-source.test.mjs`에 새 확장자 인식 테스트를 추가합니다.

## 결과를 확인하는 가장 빠른 방법

시스템에 폰트가 있다면(리눅스 기준 `/usr/share/fonts/opentype/noto/` 등) 아래처럼
실제로 한 번 돌려서 JSON을 눈으로 확인하는 게 제일 빠릅니다.

```bash
npm run build
node ./dist/bin/hwpkit-font-meta.js analyze /usr/share/fonts/opentype/noto/NotoSansCJK-Regular.ttc \
  --face NotoSansCJKkr-Regular
cat result/analyze-*.json | head -50
```

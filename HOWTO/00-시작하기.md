# HOWTO 시작하기

이 폴더는 `hwpkit-font-metadata` 코드가 어떻게 구성되어 있는지, 그리고 자주 필요한 수정을
어디서 어떻게 하면 되는지 설명합니다. 신입·주니어 개발자가 소스를 처음 열어도 따라올 수
있도록 최대한 쉬운 말로 썼습니다. 이미 코드에 익숙하다면 [../README.md](../README.md)의
사용법과 스키마 설명이 더 정확한 레퍼런스입니다.

## 이 프로젝트가 하는 일 (한 문장)

한글 폰트 파일(.ttf/.otf/.ttc 등)을 읽어서, 글자 폭·공백·줄 높이 같은 실측값을 JSON으로
뽑아내는 **로컬 CLI 도구**입니다. 이 JSON은 Hwpkit(HWP/HWPX 변환기)이 문단 줄바꿈을 더
정확히 계산하는 데 쓰입니다. 이 저장소는 JSON만 만들며, HWP/HWPX 파일이나 Hwpkit 본체를
직접 고치지 않습니다.

## 폴더 구조 한눈에 보기

```
bin/hwpkit-font-meta.ts   CLI 진입점. "analyze / compare / merge" 명령을 해석하고 실행.
src/
  font-source.ts          폰트 파일을 찾고(discover) 열어서(load) fontkit 객체로 만듦.
  analyze.ts               fontkit 객체 하나 → 글자 폭/coverage/줄 높이 등을 담은 JSON(프로필).
  compare.ts               프로필 두 개를 비교해서 거리(distance)와 점수(score)를 계산.
  stats.ts                 평균/중앙값/표준편차 같은 통계 계산 유틸리티.
  constants.ts              분석에 쓰는 코드포인트 집합, 기본 예문(corpus), 버전 상수.
  index.ts                  다른 패키지가 라이브러리로 import할 때 쓰는 공개 API 모음.
schema/                     analyze/compare 결과 JSON의 공식 스키마(JSON Schema) 문서.
examples/hwpkit-consumer.mjs  Hwpkit 쪽에서 이 JSON을 어떻게 읽어 쓰는지 보여주는 예제 어댑터.
test/                        node:test로 작성된 자동 테스트.
result/                      (자동 생성) -o 없이 실행했을 때 JSON이 쌓이는 폴더. git에 안 올라감.
```

핵심 흐름은 이렇습니다.

```
font-source.ts (파일 찾기·열기)
        │
        ▼
  analyze.ts (폰트 하나 분석 → FontProfile)
        │                     │
        ▼                     ▼
  bin/*.ts (여러 개 모아 catalog)   compare.ts (두 프로필 비교 → CompareResult)
        │
        ▼
   result/*.json 또는 -o로 지정한 파일
```

## 빌드하고 실행하기

이 프로젝트는 TypeScript로 작성되어 있고(`bin/`, `src/`), 실제로 실행되는 것은
`npm run build`가 `dist/`에 만들어내는 컴파일된 JavaScript입니다. **`src/*.ts`나
`bin/*.ts`를 고쳤다면 반드시 다시 빌드해야** `dist/`에 반영됩니다.

```bash
npm ci            # 의존성 설치 (최초 1회)
npm run build      # TypeScript → dist/ 로 컴파일
npm test           # 빌드 후 test/ 아래 테스트 전부 실행 (pretest가 자동으로 build를 돌림)
```

```bash
node ./dist/bin/hwpkit-font-meta.js --help
node ./dist/bin/hwpkit-font-meta.js analyze ./fonts/본문체.ttf
```

## 수정 → 확인까지의 기본 루틴

1. `src/` 또는 `bin/`의 `.ts` 파일을 고친다.
2. `npm run build`로 컴파일한다. 타입 에러가 나면 여기서 바로 알 수 있다.
3. `npm test`로 기존 테스트가 깨지지 않는지 확인한다.
4. 새 기능을 추가했다면 `test/*.test.mjs`에 테스트를 같이 추가한다 (기존 테스트 파일의
   패턴을 그대로 따라 하면 된다).
5. 필요하면 실제 폰트로 CLI를 한 번 더 손으로 실행해서 결과 JSON을 눈으로 확인한다.

## 자주 헷갈리는 것

- **`dist/`를 직접 고치지 마세요.** `npm run build`가 실행될 때마다 통째로 덮어씌워집니다.
  항상 `src/`와 `bin/`의 `.ts` 파일을 고치세요.
- **em 단위 vs font unit(raw) 단위.** 코드 전체에서 `...Du`로 끝나는 값은 폰트 고유의
  design unit(raw)이고, `...Em`으로 끝나는 값은 `unitsPerEm`으로 나눠서 정규화한 값입니다.
  자세한 계산식은 README의 "raw와 em을 혼동하지 않기" 절을 보세요.
- **재현 가능성(reproducibility).** 이 도구는 실행 시각이나 로컬 절대경로를 JSON 내용에
  절대 남기지 않습니다(README 참고). `result/` 폴더의 **파일 이름**에는 타임스탬프가
  들어가지만, 이건 파일을 구분하기 위한 것일 뿐 JSON 안의 내용과는 무관합니다.
